# Implementation Plan: Non-Ledger General Payments

**Branch**: `061-non-ledger-general-payments` | **Date**: 2026-05-04 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/061-non-ledger-general-payments/spec.md`

## Summary

User-defined payment categories (those outside "Fees", "Transport", "Transport + Fees") must be stored as general/non-ledger payments that do not reduce a student's outstanding balance or appear in any ledger calculation. The system also gains multi-category payment recording with per-category amount allocation and a shared receipt. A new `is_general_payment` boolean column on `payments` is the authoritative marker; `LedgerService` filters it out from all balance queries. `PaymentController` classifies category type at insert time via the existing `PaymentCategories::isSystemName()` helper.

## Technical Context

**Language/Version**: PHP 8.1 (backend) · TypeScript / React 18 (frontend)
**Primary Dependencies**: CodeIgniter 4 · MySQL · TanStack React Query · shadcn/ui · React Hook Form + Zod
**Storage**: MySQL — `payments` table (additive schema change: 2 new columns)
**Testing**: PHPUnit integration tests (`backend/tests/Integration/`)
**Target Platform**: Linux server (backend) · Browser SPA (frontend)
**Project Type**: Web service (REST API) + React SPA
**Performance Goals**: No new N+1 queries introduced; `getAllBalances` subquery pattern preserved
**Constraints**: Backward-compatible with all existing single-category payment flows; no balance regression for system-category payments
**Scale/Scope**: Touches `LedgerService` (6 query sites), `PaymentController`, `ReceiptController`, `RecordPaymentModal`, `ReceiptDocument`
