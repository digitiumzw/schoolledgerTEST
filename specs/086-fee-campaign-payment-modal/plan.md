## Summary

Add a "Fee Campaign Payment" toggle to the existing Record Payment modal. When enabled, the modal displays a dropdown of active fee campaigns. The admin selects a campaign; if the student is not yet enrolled, the frontend auto-enrolls them via `POST /api/fee-campaigns/{id}/students`, then records the payment via `POST /api/fee-campaigns/{id}/record-payment`. If already enrolled, it records payment directly. All required backend endpoints already exist from Feature 059. No schema changes needed.

## Implementation Approach

### Frontend-Only Changes

1. **State additions to `RecordPaymentModal.tsx`**:
   - `campaignPaymentMode: boolean` — toggles campaign payment flow
   - `campaigns: FeeCampaign[]` — all active campaigns for the tenant
   - `studentCampaigns: StudentCampaignMembership[]` — student's existing memberships
   - `selectedCampaignId: string | null` — chosen campaign
   - `campaignsLoading: boolean` — loading state for campaign fetch
   - `isEnrolledInSelected: boolean` — derived from comparing selectedCampaignId with studentCampaigns

2. **Data fetching**:
   - When `campaignPaymentMode` is enabled AND a student is selected:
     - Fetch `api.getFeeCampaigns({ status: 'active', limit: 100 })`
     - Fetch `api.getStudentCampaigns(studentId)`
     - Both calls fire concurrently
   - Use `useEffect` with cleanup to prevent race conditions when student changes

3. **UI changes**:
   - Add a Switch/checkbox labeled "Fee Campaign Payment" below the method selector
   - When active: hide category selector and multi-category toggle; show campaign dropdown
   - Campaign dropdown items show:
     - Campaign name
     - "Member" badge (if student is already enrolled)
     - Expected amount / Paid so far / Remaining balance (for member campaigns)
   - Amount input pre-fills with `remainingAmount` when a campaign is selected

4. **Submission flow**:
   - Validate: student selected, campaign selected, amount > 0, method selected
   - Check `isEnrolledInSelected`:
     - If `false` → call `api.addCampaignStudent(selectedCampaignId, studentId)` first
       - On success → call `api.recordCampaignPayment(selectedCampaignId, payload)`
       - On error → show toast error, keep modal open, do NOT attempt payment
     - If `true` → call `api.recordCampaignPayment(selectedCampaignId, payload)` directly
   - `submitting` flag covers the entire sequence; button disabled during both calls
   - On success → invalidate caches, show success toast, close modal, trigger receipt
   - On any error → show error toast, keep modal open

5. **Cache invalidation on success**:
   - `['student-balance', studentId]`
   - `['payments-with-students']`
   - `['dashboard', 'activity']`
   - `['fee-campaigns']`
   - `['student-campaigns', studentId]`

### Backend: No Changes Required

All endpoints are fully functional from Feature 059:
- `GET /api/fee-campaigns?status=active&limit=100` — paginated active campaigns with summary
- `GET /api/students/:id/campaigns` — student's campaign memberships with amounts
- `POST /api/fee-campaigns/:id/students` — auto-enroll student (returns 201 or 400 if already enrolled)
- `POST /api/fee-campaigns/:id/record-payment` — record payment (returns 201 or error)

The existing `FeeCampaignService::recordPayment` already:
- Validates campaign is active (returns 409 if closed)
- Validates student is enrolled (returns 404 if not)
- Validates amount bounds (returns 400 if exceeds remaining or <= 0)
- Runs in a database transaction
- Updates `campaign_students.paid_amount` and `status`
- Creates a `payments` row with `fee_campaign_id` and snapshot JSON

## Risk Mitigation

| Risk | Mitigation |
|------|------------|
| Race condition: campaign closes between enroll and pay | Backend `recordPayment` already checks campaign status and returns 409. Frontend surfaces error. |
| Double submission | Submit button disabled during entire in-flight sequence. |
| Partial state: enrolled but not paid | Acceptable. Admin can retry payment. Student is now in the campaign. |
| Large number of campaigns (>100) | Use `limit=100`. If more exist, consider a searchable dropdown instead of a native Select. Out of scope for v1. |
| Modal state pollution on toggle off | Reset `selectedCampaignId`, clear campaign-related state when toggling off. Standard category flow reappears clean. |
