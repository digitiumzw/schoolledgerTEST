# Implementation Plan: Backend-Driven Architecture

**Branch**: `084-backend-driven-architecture` | **Date**: 2026-05-25 | **Spec**: [spec.md](spec.md)  
**Input**: Feature specification from `specs/084-backend-driven-architecture/spec.md`

## Summary

Refactor Staff, Fee Campaigns, and Transport modules to eliminate remaining client-side
filtering, sorting, and pagination. Fix the N+1 per-campaign summary query in
FeeCampaignController. Migrate these three modules from raw `useEffect + useState` to
React Query. Add global `refetchInterval: 30_000` to the QueryClient for automatic
real-time data polling. Optionally add a database migration for performance indexes on
the `staff` table's filter columns.

## Technical Context

**Language/Version**: PHP 8.1+ (backend) · TypeScript 5.x / React 18 (frontend)  
**Primary Dependencies**: CodeIgniter 4 (backend) · TanStack React Query v5 · shadcn/ui · TailwindCSS  
**Storage**: MySQL — `staff`, `fee_campaigns`, `campaign_students`, `transport_routes`, `transport_vehicles`, `transport_drivers` tables  
**Testing**: curl-based integration tests (Constitution Principle X)  
**Target Platform**: Web SPA (React) + REST API (PHP/CodeIgniter 4)  
**Project Type**: Multi-tenant SaaS web application  
**Performance Goals**: Staff list ≤ 500ms; Transport list ≤ 400ms; Fee Campaigns list ≤ 300ms; all at expected data volumes  
**Constraints**: Max 100 records per page; ≤ 3 SQL queries per paginated list request; no client-side filtering/sorting/pagination  
**Scale/Scope**: Per-tenant staff typically 50–500; campaigns typically < 50; transport typically < 100 routes

## Constitution Check

*Pre-design gate: PASS. Post-design gate: PASS.*

**Principle I — Multi-Tenant Data Isolation**: All new `StaffModel::getFiltered()` queries,
`FeeCampaignModel::getSummariesByCampaignIds()`, and transport pagination queries filter by
`tenant_id` sourced from JWT. No tenant_id accepted from request params. ✅

**Principle II — API-First Separation**: Frontend receives only paginated, filtered,
pre-computed data from backend APIs. No business logic in frontend pages or hooks. ✅

**Principle III — JWT Auth & RBAC**: All affected endpoints are already behind `JWTAuthFilter`.
No new public routes. ✅

**Principle IV — Immutable Migrations**: One optional new migration file
`2026-05-25-000001_AddStaffFilterIndexes.php` for staff table indexes. No existing migrations
edited. ✅

**Principle V — Financial Ledger Integrity**: No ledger queries affected by this feature. ✅

**Principle VI — REST Standards**: No new endpoints; existing endpoint paths unchanged.
Response shape update (pagination wrapper) follows standard envelope via `respondSuccess`. ✅

**Principle VII — Code Quality**: Client-side filter/sort/slice logic deleted, not retained.
Paginated query helpers extracted to model methods. ✅

**Principle VIII — Defensive Security**: All pagination params validated (range checks, enum
whitelist for sortBy). Invalid params return HTTP 400. ✅

**Principle IX — Error Handling**: Validation errors return standard error envelope. No internal
query details exposed. ✅

**Principle X — curl Tests**: `quickstart.md` defines 12 curl validation scenarios covering
happy paths, error paths, and tenant isolation. ✅

**Principle XI — Backend-Driven Data & Performance**:
- Staff: `getFiltered()` uses indexed columns; pagination applied at SQL layer; no N+1
- Fee Campaigns: `getSummariesByCampaignIds()` replaces N separate queries with one GROUP BY; bounded by campaign list
- Transport: LIMIT/OFFSET applied before IN-query enrichment; enrichment still batch-safe
- React Query `refetchInterval` + `staleTime` aligned at 30s for all active queries ✅

**Principle XII — Mutation Loading States**: All three migrated pages use React Query `useQuery`
+ `useMutation`; hooks expose `isPending`; `queryClient.invalidateQueries` called on all
create/update/delete mutations in these modules. ✅

**Principle XIII — Email Design System**: No email changes in this feature. N/A ✅
