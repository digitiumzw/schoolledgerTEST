# Implementation Plan: Performance & Scalability Optimization

**Branch**: `066-performance-scalability-optimization` | **Date**: 2026-05-07 | **Spec**: [spec.md](./spec.md)  
**Input**: Feature specification from `specs/066-performance-scalability-optimization/spec.md`

## Summary

Eliminate client-side data processing across five heavy-data pages (Payments, Attendance, Transport, ClassStudentsPage, Dashboard) by moving filtering, sorting, aggregation, and pagination to the database layer. The primary change is refactoring `PaymentController::withStudents()` to accept pagination/filter/sort params and return a standardised `{ data, pagination, stats }` envelope — removing the `GET /ledger/balances` call from the Payments page. Secondary changes add a new `GET /student-attendance/class-summary` backend endpoint, server-side `search` params to transport and class endpoints, and replace the O(n) PHP student loop in `DashboardController::stats()` with a SQL aggregate query.

## Technical Context

**Language/Version**: PHP 8.1 · TypeScript (strict mode)  
**Primary Dependencies**: CodeIgniter 4 · React 18 · TanStack React Query v5 · TailwindCSS · shadcn/ui  
**Storage**: MySQL  
**Testing**: curl (Principle X) · PHP lint · TypeScript `tsc --noEmit`  
**Target Platform**: Linux server (backend) · Browser SPA (frontend)  
**Project Type**: Web application (REST API + React SPA)  
**Performance Goals**: Payments page initial load payload ≤ 20 records; no full-tenant balance scan on page load; all filter/search changes ≤ 1s under normal network conditions  
**Constraints**: No schema migrations required; existing route paths must remain unchanged; `GET /ledger/balances` must not be deleted  
**Scale/Scope**: Schools with 50–2,000 students; up to 10,000 payment records per tenant
