# Implementation Plan: Student Attendance – Class-Linked Event Tracking

**Branch**: `068-student-attendance-classes` | **Date**: 2026-05-08 | **Spec**: [spec.md](spec.md)  
**Input**: Feature specification from `/specs/068-student-attendance-classes/spec.md`

## Summary

Replace the simple upsert-based `student_attendance` row model with an additive immutable-event system (`student_attendance_events`) that ties every attendance mark to a `class_instance_id`, `academic_session`, and `submitted_by` user. A new `StudentClassAttendanceService` handles batch submission (with enrollment validation, future-date guard, and effective-flag cascading), aggregation queries, and audit-log reads. A new `StudentClassAttendanceController` exposes six REST endpoints. The existing `student_attendance` table and all kiosk/legacy endpoints remain untouched. Per-period mode is toggled via the existing settings JSON config. The React frontend adds a "Class Attendance" tab to `Attendance.tsx` backed by a new `useClassAttendance.ts` hook.

## Technical Context

**Language/Version**: PHP 8.1+ · TypeScript 5  
**Primary Dependencies**: CodeIgniter 4 (backend) · React 18 + TanStack Query + shadcn/ui (frontend)  
**Storage**: MySQL — new `student_attendance_events` table; existing `class_instances`, `enrollments`, `students`, `settings` tables referenced  
**Testing**: curl-based endpoint tests after implementation (Constitution Principle X)  
**Target Platform**: Linux server (backend) · Browser SPA (frontend)  
**Project Type**: Web application (REST API + React SPA)  
**Performance Goals**: Batch of 50 students submits in < 3 s (SC-001); aggregation queries < 500 ms for up to 1 academic year of data  
**Constraints**: All queries prefixed with `tenant_id` (Principle I); no mutable update/delete on event rows; `is_effective` cascade in a single DB transaction  
**Scale/Scope**: ~30 classes per tenant; ~50 students per class; ~200 school days per year → ~300,000 rows/tenant/year max

## Key Design Decisions (from research.md)

1. **New table `student_attendance_events`** — additive; existing `student_attendance` table untouched.
2. **`class_instance_id` as primary link** — ties every event to the class × academic year intersection.
3. **Immutability via `is_effective` flag** — service cascades the flag in a transaction; no DB triggers needed.
4. **`period_key` nullable** — NULL = per-day; string = per-period; single-table model for both modes.
5. **`studentAttendanceMode` setting** — added to existing settings JSON config (no schema migration for settings).
6. **`StudentClassAttendanceController`** — dedicated controller; `AttendanceController` not touched.
7. **Partial-success batch response** — non-enrolled students produce `skipped` entries, not a 422 error.
8. **`academic_session` denormalized** — avoids JOIN to `class_instances` on every aggregation query.

## MVP Scope

- **Phase 1 + Phase 2 (US1)**: Migration + model + service + controller + routes + frontend batch submit + read. Delivers the core daily recording workflow.
- **Phase 3 (US2 per-period + US3 reporting + US4 audit)**: Per-period mode, aggregation endpoints, audit log, settings UI toggle.
