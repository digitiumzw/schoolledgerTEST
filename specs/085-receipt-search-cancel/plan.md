# Implementation Plan: Receipt Search and Cancel

**Branch**: `085-receipt-search-cancel` | **Date**: 2026-05-30 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/085-receipt-search-cancel/spec.md`

## Summary

Add a dedicated receipt-number search field to the Payments page and implement a cancel/void action for receipts. When a receipt is canceled, the associated payment rows are soft-voided (voided_at timestamp + reason) and the student's ledger balance is recalculated to exclude the voided amount. The receipt view displays a clear "CANCELED / INVALID" indicator for voided receipts. No new database schema is required — `receipt_number`, `voided_at`, `void_reason`, and `voided_by` already exist on the `payments` table.

## Technical Context

**Language/Version**: PHP 8.1+ · CodeIgniter 4  
**Primary Dependencies**: MySQL, JWTAuthFilter, BaseApiController  
**Storage**: MySQL (existing `payments` table with `receipt_number`, `voided_at`, `void_reason`, `voided_by`)  
**Testing**: curl URL requests (Principle X), PHP lint  
**Target Platform**: Linux server (backend), Web browser (React SPA frontend)  
**Project Type**: Web application (backend + frontend SPA)  
**Performance Goals**: Receipt search returns paginated results in < 300ms p95 at expected volume; void + ledger recalculation completes in < 3s  
**Constraints**: All queries must include `tenant_id` (Principle I); no client-side filtering/sorting/computation (Principle XI); balance always computed from source records (Principle V)  
**Scale/Scope**: Multi-tenant SaaS; expected payment volume 10k–50k per tenant

| Principle | Status | Notes |

| III. JWT & Role-Based Access | **PASS** | Void endpoint requires `admin` or `bursar` role. Search endpoint requires authenticated user (any role with payments access). |

| V. Ledger Integrity | **PASS** | Voiding triggers `LedgerService::getStudentBalance()` which recomputes from source records. Pre-existing gap: payment subqueries in `LedgerService` do not yet filter `voided_at IS NULL` — this will be fixed as part of this feature. |
| VI. REST Standards | **PASS** | `POST /api/payments/:id/void` follows REST action-on-resource pattern. Response uses `BaseApiController` helpers. |
| VII. Code Quality | **PASS** | Small, focused changes to existing controllers/models. No dead code introduced. |
| VIII. Defensive Security | **PASS** | Input sanitized (`sanitiseString` for reason). Role check in controller. Tenant isolation enforced. |

| X. API Endpoint Testing | **PASS** | curl tests defined in quickstart.md: happy path, already-voided error, unauthorized role, tenant isolation, receipt search, voided receipt view. |
| XI. Backend-Driven Data | **PASS** | Search/filter/pagination already server-side. Void action is server-side with ledger recalculation. Frontend only renders backend-prepared responses. |
| XII. Mutation Loading States | **PASS** | Void mutation hook (`useCancelReceipt`) exposes `isPending`. Payment list query invalidated after void. Action button disabled during in-flight request. |
