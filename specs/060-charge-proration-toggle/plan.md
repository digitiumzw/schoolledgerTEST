# Implementation Plan: Charge Proration Toggle

**Branch**: `060-charge-proration-toggle` | **Date**: 2026-05-04 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/060-charge-proration-toggle/spec.md`

## Summary

Add a per-tenant `chargeProrationEnabled` boolean toggle (default `false`) to the Settings / Fee Structure section. When enabled, both the fee-rule billing engine (`FeeRuleBillingService::generateCharges`) and the transport monthly charge generator (`TransportController::generateMonthlyCharges`) calculate a prorated amount for students whose `enrollment_date` (fee rules) or transport allocation `start_date` (transport) falls after the billing period start. A new `ChargeProrationHelper` service encapsulates the `floor(remaining/total × amount)` formula, preventing duplication across both generators. No database migration is required; the setting is stored in the existing `tenants.settings` JSON column. The `/settings/fee-structure` route gains a new Card with a Switch toggle; the settings sidebar gains a corresponding entry.

## Technical Context

**Language/Version**: PHP 8.1 (backend) · TypeScript / React 18 (frontend)
**Primary Dependencies**: CodeIgniter 4 · MySQL · Vite · TailwindCSS · shadcn/ui
**Storage**: `tenants.settings` JSON column (existing) — no new tables or columns
**Testing**: PHPUnit integration tests (`backend/tests/Integration/`)
**Target Platform**: Linux server (backend) · Browser SPA (frontend)
**Project Type**: Web application (REST API + React SPA)
**Performance Goals**: No new N+1 queries; proration math is pure in-memory arithmetic per student row already fetched
**Constraints**: Zero migrations (FR-014); backward-compatible default (`false`); no change to existing charge output when toggle is off
